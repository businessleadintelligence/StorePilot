Prisma Migrate — version-controlled, sequential migration files

Supabase PgBouncer (transaction mode for serverless)

Encryption
Column-level AES-256 for accessToken and ga4RefreshToken; Supabase at-rest encryption for all data
Total Core Tables

1.1 Design Principles
Multi-tenancy: Every table containing merchant data includes a storeId foreign key. No cross-store query is possible without explicitly crossing tenant boundaries.
Soft-delete over hard-delete: Anomalies, recommendations, and timeline events are never deleted — they form the operational memory moat.
Append-only metrics: Daily metrics tables grow by insertion, never update. Historical accuracy is preserved.
Deterministic-first: The database stores all pre-computed deterministic values (scores, impact figures). The AI layer reads these values; it never computes them.
Normalization balanced with query performance: Denormalization is applied deliberately in high-frequency read paths (e.g., storeId on all child tables for direct index access).

4.2 Retention Enforcement
A nightly scheduled job (cron) identifies rows in time-series tables that exceed the store's plan retention window.
Rows are hard-deleted from metrics tables after the retention window — they cannot be recovered.
Anomalies, recommendations, and timeline events are NEVER purged as they constitute the operational memory moat.
When a store upgrades from Starter to Growth, existing data within the new window is immediately accessible — no data migration required.
When a store downgrades, data outside the new window is flagged for deletion at the next nightly purge cycle.
4.3 Store Deletion & GDPR
When a store uninstalls the app (app/uninstalled webhook), the store is marked active = false. Data is not immediately deleted.
If a store requests full data deletion (GDPR erasure request), a manual process cascades DELETE across all tables for that storeId.
The stores.accessToken and ga4RefreshToken columns are zeroed immediately on uninstall, regardless of data retention status.

5.2 Baseline Calculation
Anomaly detection requires a baseline to compare against. Baselines are calculated from historical data at import time and recalculated on each daily sync:
Revenue baseline: 30-day rolling average daily revenue
Conversion baseline: 30-day rolling average conversion rate
Refund baseline: 30-day rolling average refund rate
Traffic baseline: 30-day rolling average daily sessions (when GA4 connected)
Inventory baseline: 14-day rolling average daily units sold per product
Stores with fewer than 14 days of history at onboarding receive reduced confidence scores (Moderate or Low) until sufficient baseline data accumulates. The confidence engine accounts for this automatically.

6.2 AI COO Impact Score Calculation
The AI COO Impact Score is a cumulative rollup across multiple tables. It is calculated on-demand and optionally cached in the reports table:
-- Revenue Protected: sum of revenueImpact for all resolved anomalies
SELECT SUM(a.revenueImpact)
  FROM anomalies a
  WHERE a.storeId = $1 AND a.resolvedAt IS NOT NULL;
-- Revenue Opportunity Identified: sum of all recommendation revenueImpact
SELECT SUM(r.revenueImpact)
  FROM recommendations r
  WHERE r.storeId = $1;
-- Recommendations Completed: count of completed recommendations
SELECT COUNT(*)
  FROM recommendations r
  WHERE r.storeId = $1 AND r.status = 'completed';
-- Problems Resolved: count of resolved anomalies
SELECT COUNT(*)
  FROM anomalies a
  WHERE a.storeId = $1 AND a.resolvedAt IS NOT NULL;

7.2 Context Injection Query
Before every AI call, the following context retrieval query is executed to inject relevant operational history into the prompt:
-- Step 1: Last 3 anomalies of the same type (for pattern context)
SELECT a.type, a.metricValue, a.baselineValue, a.triggeredAt,
       a.resolvedAt, ac.causeText
  FROM anomalies a
  LEFT JOIN anomaly_causes ac ON ac.anomalyId = a.id
  WHERE a.storeId = $1 AND a.type = $2
  ORDER BY a.triggeredAt DESC
  LIMIT 3;
-- Step 2: Last 5 recommendations for this anomaly type and their outcomes
SELECT r.problem, r.action, r.status, r.completedAt,
       ro.anomalyResolved, ro.revenueImpactActual
  FROM recommendations r
  LEFT JOIN recommendation_outcomes ro ON ro.recommendationId = r.id
  JOIN anomalies a ON a.id = r.anomalyId
  WHERE r.storeId = $1 AND a.type = $2
  ORDER BY r.createdAt DESC
  LIMIT 5;
-- Step 3: Recent Business Timeline (last 30 days)
SELECT eventType, occurredAt, title, revenueImpact
  FROM timeline_events
  WHERE storeId = $1
    AND occurredAt >= NOW() - INTERVAL '30 days'
  ORDER BY occurredAt DESC
  LIMIT 20;
7.3 Proprietary Intelligence Asset
The recommendation_outcomes table grows as merchants complete recommendations. This dataset enables future capabilities described in the Founder Specification:
Aggregate outcome accuracy measurement: what percentage of recommendations lead to anomaly resolution
Revenue impact accuracy: estimated vs actual impact differential over time
Cross-store pattern learning: which recommendation types have the highest completion and resolution rates by store type or revenue tier
These cross-store analytics capabilities are post-V1 roadmap items. In V1, recommendation_outcomes data is used only for per-store outcome display and AI COO Impact Score calculation. No cross-store data access or ML training pipeline is built in V1.
