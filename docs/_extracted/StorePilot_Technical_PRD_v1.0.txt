6.2 Shopify OAuth Flow
SHOPIFY OAUTH FLOW
1. Merchant clicks Install from Shopify App Store
   GET /api/shopify/oauth/initiate?shop={shop}.myshopify.com
2. Server validates shop param, builds OAuth URL:
   https://{shop}.myshopify.com/admin/oauth/authorize
     ?client_id={SHOPIFY_API_KEY}
     &scope={REQUIRED_SCOPES}
     &redirect_uri={APP_URL}/api/shopify/oauth/callback
     &state={nonce}  // CSRF token stored in session
3. Merchant approves on Shopify consent screen
4. Shopify redirects to:
   GET /api/shopify/oauth/callback?shop=...&code=...&state=...&hmac=...
5. Server validates:
   a. state matches stored nonce (CSRF check)
   b. HMAC signature valid
   c. shop param matches expected format
6. Server exchanges code for permanent access token:
   POST https://{shop}.myshopify.com/admin/oauth/access_token
     { client_id, client_secret, code }
7. Token stored encrypted in Stores.accessToken
8. Server creates Store record and User record
9. Background job triggered: historical data import (90 days)
10. Merchant redirected to /onboarding

6.4 Session Lifecycle
Sessions are validated on every protected API route and page. Token refresh is handled automatically. If a Shopify token is revoked (uninstall webhook), the store record is marked inactive and all sessions for that store are invalidated.

7.4 Webhook Validation
WEBHOOK HMAC VALIDATION
// Every webhook handler MUST call this before processing
function validateShopifyWebhook(rawBody: Buffer, hmacHeader: string): boolean {
  const hash = crypto
    .createHmac('sha256', process.env.SHOPIFY_WEBHOOK_SECRET)
    .update(rawBody)
    .digest('base64');
  return crypto.timingSafeEqual(Buffer.from(hash), Buffer.from(hmacHeader));
}
// Return 401 immediately if validation fails. Never process unauthenticated payloads.

9.2 AI Responsibility Boundary
The AI layer NEVER calculates. It receives structured, pre-computed data from deterministic engines and its ONLY job is to generate natural-language text. Any AI response that contains a number must have received that number as input from a deterministic engine, never inferred it.
9.3 Prompt Architecture
Each AI feature uses a structured prompt template. All prompts follow the same three-layer pattern:
PROMPT TEMPLATE STRUCTURE
SYSTEM PROMPT (constant per feature):
  - Defines AI role and constraints
  - Enforces business language requirements
  - Specifies confidence level handling rules
  - Instructs AI never to fabricate numbers
CONTEXT BLOCK (dynamic — injected per request):
  - Pre-computed metrics (health score, revenue at risk, etc.)
  - Active anomalies with computed severity
  - Relevant operational history from memory system
  - Confidence level for this recommendation
INSTRUCTION BLOCK (dynamic — specifies output format):
  - Problem: [describe detected issue]
  - Likely Cause: [explain why this is happening]
  - Revenue Impact: [use the provided figure: {revenue_impact}]
  - Recommended Action: [specific, actionable step]
  - Confidence: [use the provided level: {confidence_level}]

9.5 Operational Memory Integration
The Operational Memory System feeds historical context into AI prompts. Before any AI call, the system retrieves relevant history from the database:
Last 3 similar anomalies and their resolutions
Last 5 recommendations for this anomaly type and their outcomes
Key business events from Business Timeline (last 30 days)
Previous AI COO Chat context (last 10 messages for chat endpoint)
This context is injected into the CONTEXT BLOCK of each prompt to enable progressively more relevant explanations as the store's history grows.

11.3 Job Execution Pattern
CRON JOB PATTERN
// Pattern for all cron handlers
export async function GET(req: Request) {
  // 1. Validate cron secret
  if (req.headers.get('Authorization') !== `Bearer ${process.env.CRON_SECRET}`) {
    return new Response('Unauthorized', { status: 401 });
  }
  // 2. Fetch all active stores
  const stores = await db.stores.findMany({ where: { active: true } });
  // 3. Process each store with error isolation
  const results = await Promise.allSettled(
    stores.map(store => processStore(store).catch(err => {
      logger.error({ storeId: store.id, err }, 'Cron job failed for store');
      // Never throw — one store failure must not block others
    }))
  );
  return Response.json({ processed: stores.length, results });
}

13. Error Handling Strategy
13.1 Error Handling Principles
Store-level isolation: A failure for one merchant must never affect other merchants
Graceful degradation: Partial data is always better than no data
Retry with backoff: All external API calls implement exponential backoff with jitter
Structured logging: All errors logged with storeId, operation, and error context
User-facing errors: Never expose internal error details; show actionable merchant messages
13.2 Error Categories & Responses
Error Category
Example
Handling Strategy
Shopify API Rate Limit (429)
Too many GraphQL requests
Exponential backoff: 1s, 2s, 4s, 8s — max 3 retries. Log and skip store on final failure.
Shopify API Unavailable (5xx)
Shopify maintenance window
Retry 3x with backoff. Mark sync as failed; next scheduled run will catch up via delta sync.
Webhook HMAC Failure
Forged or malformed webhook
Return 401 immediately. Log source IP. Never process payload.
OpenAI API Error
Rate limit, model unavailable
Return cached last briefing if available. Log and alert. Never show blank briefing.
GA4 Token Expired
refresh_token invalid
Mark GA4 as disconnected for store. Disable GA4-dependent features. Notify merchant via email.
Database Connection Error
Supabase connection pool exhausted
Retry 2x. Return 503 to client. Vercel will retry scheduled jobs.
Cron Job Partial Failure
3 of 50 stores fail during hourly sync
Log per-store failures. Continue processing remaining stores. Success metric: % stores synced.
Invalid Merchant Session
Expired or revoked session token
Redirect to /auth/shopify for re-authentication.

  }
  // Process new order...
}

15.2 Cron Job Scaling Design
As the merchant base grows, the hourly cron job must process all active stores within 60 minutes. Architecture must support this growth:
Phase 1 (< 500 stores): Single cron invocation, sequential or parallel Promise.allSettled
Phase 2 (500–2,000 stores): Fan-out pattern — cron invokes N parallel worker functions
Phase 3 (2,000+ stores): Queue-based processing (Vercel Queue or external queue) — outside V1 scope
