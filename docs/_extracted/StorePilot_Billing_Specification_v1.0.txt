SECTION 1  ·  Billing Overview & Principles
1. Billing Overview & Principles
1.1  Purpose
This specification defines the complete billing architecture for StorePilot. It covers subscription lifecycle management, trial mechanics, plan transitions, failure handling, database schema, webhook integration, API endpoints, acceptance criteria, edge cases, security, and future scalability. All billing is exclusively managed through Shopify's App Subscriptions API. StorePilot must not implement independent payment processing.
1.2  Guiding Principles
Shopify is the single source of billing truth. StorePilot's database reflects Shopify's billing state; it never overrides it.
No money is collected outside Shopify. StorePilot never handles card details or initiates charges directly.
Merchant experience must be frictionless. Trial activation, plan changes, and cancellations complete in under 60 seconds.
Billing state changes are atomic. Partial state writes must never leave the application in an inconsistent billing state.
Revenue is protected through automated retry logic and transparent merchant communication on payment failure.
The 14-day free trial requires no credit card. Merchants evaluate StorePilot risk-free.
1.3  Shopify Managed Billing — Architecture Summary
StorePilot uses Shopify's RecurringApplicationCharge (REST) and AppSubscription (GraphQL Admin API) primitives. All pricing, trial logic, and charge collection are delegated to Shopify. StorePilot stores billing metadata in its own PostgreSQL database (hosted on Supabase) to drive feature entitlements, usage enforcement, and reporting.
CRITICAL
StorePilot must never implement its own payment gateway, store raw card data, or bypass Shopify Billing. Violations create legal liability and violate Shopify Partner Program terms.

2.3  Usage Limits & Enforcement
2.3.1  AI Chat Message Limits
AI Chat message counts are tracked per billing cycle per store. The billing cycle resets at the start of each Shopify billing period. When a merchant reaches 90% of their monthly limit, a warning is displayed in the AI COO Chat interface. At 100%, the chat input is disabled and a plan upgrade prompt is shown. Message counts are stored in the StorePilot database and are not enforced by Shopify.
2.3.2  Store Limits
Starter and Growth plans are hard-limited to 1 store. Agency plans allow up to 5 stores. Attempts to connect additional stores beyond the plan limit return a 403 error with an upgrade prompt. Store count is enforced at the application layer.
2.3.3  Historical Data Access
Data older than the plan's history window is retained in the database but excluded from all queries, dashboards, and AI context for merchants on plans with shorter history windows. Upgrading immediately unlocks access to all stored historical data. Data is never deleted on downgrade; it is only masked.
SECTION 3  ·  Subscription Lifecycle
3. Subscription Lifecycle
3.1  Trial Lifecycle
3.1.1  Trial Initiation
When a merchant installs StorePilot and completes OAuth, the application creates an AppSubscription via Shopify GraphQL with a trialDays value of 14 and the merchant's selected plan (defaulting to Starter if no plan is explicitly chosen during onboarding).

3.1.3  Trial Rules
Trial duration is 14 calendar days from merchant approval of the subscription on Shopify's confirmation page.
No credit card is required to start the trial. Shopify handles payment method collection when the trial ends.
Trial entitlements are identical to the selected paid plan. Full feature access is granted during the trial.
A merchant who uninstalls during trial and reinstalls within the same Shopify billing context does not receive a second 14-day trial. Refer to Section 3.6 for reinstallation logic.
Trial period cannot be extended by the application. Extensions require manual Shopify Partner Dashboard intervention.

3.3  Upgrade Flow
3.3.1  Merchant-Initiated Upgrade
Merchant selects a higher-tier plan in StorePilot settings.
StorePilot backend calls AppSubscription create via Shopify GraphQL with the new plan.
Shopify generates a confirmation URL and returns it to StorePilot.
StorePilot redirects the merchant to the Shopify confirmation page.
Merchant approves the new subscription on Shopify's confirmation page.
Shopify fires app_subscriptions/update webhook with status ACTIVE.
StorePilot webhook handler updates the subscriptions table with the new plan_key, shopify_subscription_id, and status.
StorePilot immediately applies new feature entitlements (increased chat limit, additional stores for Agency, extended history).
StorePilot displays an upgrade confirmation in-app and records a timeline event.
3.3.2  Upgrade Proration
Shopify handles proration automatically when upgrading mid-cycle. Merchants are charged the prorated difference for the remainder of the billing cycle. StorePilot does not calculate or display proration amounts directly; it relies on Shopify's billing calculation. The upgrade confirmation page should inform merchants that Shopify will manage proration.
3.3.3  Immediate Entitlement
Feature entitlements must be upgraded immediately upon Shopify confirming the new subscription as ACTIVE. There must be no delay between subscription approval and feature access. Chat message limits are recalculated at the point of upgrade, not at the next billing cycle reset.
3.4  Downgrade Flow
3.4.1  Merchant-Initiated Downgrade
Merchant selects a lower-tier plan in StorePilot settings.
StorePilot displays a downgrade impact summary showing features that will be lost.
Merchant confirms downgrade.
StorePilot backend calls AppSubscription create with the lower-tier plan.
Shopify generates a confirmation URL; merchant is redirected to confirm.
Merchant approves; Shopify fires app_subscriptions/update webhook.
StorePilot updates subscription record and applies lower-tier entitlements.
Timeline event is recorded. Downgrade email notification is sent.
3.4.2  Downgrade Constraints
A merchant on Agency with 3 connected stores cannot downgrade to Growth or Starter until they disconnect stores to meet the 1-store limit. StorePilot must enforce this pre-check before initiating the downgrade flow.
Historical data is not deleted on downgrade. It is masked at the query layer. Upgrading restores full access.
AI Chat message usage counts are not reset on downgrade. The remaining messages available are recalculated against the new plan limit for the remainder of the current billing cycle.
Downgrade takes effect at the start of the next billing cycle unless Shopify processes it immediately (Shopify handles timing).
3.5  Failed Payment Handling
3.5.1  Shopify Payment Retry Behaviour
Shopify automatically retries failed recurring charges on its own schedule. StorePilot does not control retry timing. Shopify will fire app_subscriptions/update webhooks as retry outcomes are determined.
3.5.2  StorePilot Response to Failed Payment

3.5.3  Grace Period
StorePilot provides a 3-day grace period from the first DECLINED webhook before restricting feature access. This prevents disruption from transient payment failures. During the grace period, the merchant sees an in-app payment failure banner with a link to their Shopify billing settings. Access restriction after 3 days means the merchant can only view the billing/account settings page.
3.5.4  Data Preservation During Failure
All store data, historical metrics, and operational memory are fully preserved during the failed payment and grace period. Data is never deleted due to payment failure. Access is restricted, not data destroyed.
3.6  Cancellation Flow
3.6.1  Merchant-Initiated Cancellation
Merchants cancel through Shopify directly (via their Shopify admin, Partner Dashboard, or StorePilot settings). StorePilot does not provide a direct cancel button but provides a link to Shopify's subscription management.
Merchant cancels through Shopify.
Shopify fires app_subscriptions/update webhook with status CANCELLED.
StorePilot webhook handler sets subscription status to CANCELLED and records cancelled_at timestamp.
Access to StorePilot features is revoked immediately.
Data is retained for 90 days post-cancellation (data retention period).
Cancellation email is sent with data retention notice.
3.6.2  Data Retention Post-Cancellation
All store data is retained for 90 days after cancellation.
If a merchant reinstalls within 90 days, their data is fully restored.
After 90 days, a scheduled job marks data for deletion. Deletion executes in batches.
Merchants are notified by email at 60 days and 85 days post-cancellation if data deletion is approaching.
3.7  Reinstallation Flow
3.7.1  Reinstall Within 90-Day Data Retention Window
Merchant reinstalls StorePilot from Shopify App Store.
StorePilot OAuth flow executes.
StorePilot checks for existing store record by shop domain.
Existing record found with status CANCELLED within retention window.
StorePilot creates a new AppSubscription with NO trial days (trial already consumed).
Merchant approves subscription; Shopify charges immediately for the billing cycle.
On ACTIVE webhook, StorePilot restores all existing store data.
In-app message confirms data restoration.
3.7.2  Reinstall After 90-Day Data Retention Window
Merchant reinstalls StorePilot.
StorePilot OAuth flow executes.
StorePilot checks for existing store record by shop domain.
Existing record found but data has been deleted or marked for deletion.
StorePilot creates a new AppSubscription with NO trial days.
Onboarding executes as a fresh install. 90-day historical data import runs.
In-app message notifies merchant that previous data was not retained.
3.7.3  Trial on Reinstall
A second 14-day trial is not granted on reinstallation regardless of circumstances. StorePilot must track trial_used boolean on the stores table. If trial_used is true, all new subscriptions for that shop domain are created with trialDays: 0.
RULE
One trial per shop domain. The shop domain is the canonical identifier. Merchant account changes or store transfers do not reset trial eligibility.

SECTION 5  ·  Required Shopify Webhooks
5. Required Shopify Webhooks
All webhooks must be registered during app installation. Shopify delivers webhooks via HTTPS POST. StorePilot must verify each webhook using HMAC-SHA256 signature validation before processing. Invalid signatures must return HTTP 401 and be logged.
5.1  Webhook Registration
Webhooks are registered during OAuth installation using the Shopify Admin API. The following table defines all required billing-related webhook topics and their StorePilot handlers.
Webhook Topic
Trigger
StorePilot Handler Action
app_subscriptions/update
Subscription status change
Primary billing state handler. Updates subscriptions table. Triggers entitlement changes, notifications, and timeline events.
app/uninstalled
Merchant uninstalls app
Sets store status to inactive. Does not delete data. Schedules data deletion if subscription is not already cancelled.
shop/update
Shop details changed
Updates store contact email and timezone. Used to keep billing notification email current.
5.2  app_subscriptions/update — Handler Logic
5.2.1  Status Routing
The webhook handler must route on the status field in the payload and execute the appropriate state transition:
Status
StorePilot Action
Notifications
ACTIVE
Set subscription status to active. Update plan entitlements. Clear grace_period_ends_at. Clear billing_access_restricted.
None (unless reinstating from DECLINED – send payment restored email)
DECLINED
Set status to declined. Set grace_period_ends_at to now + 3 days. Schedule access restriction job.
Payment failed email + in-app banner immediately
CANCELLED
Set status to cancelled. Set cancelled_at. Set data_deletion_scheduled_at to cancelled_at + 90 days. Revoke feature access.
Cancellation confirmed email with data retention notice
FROZEN
Set billing_access_restricted to true. Suspend feature access.
In-app banner only
EXPIRED
Treat identical to CANCELLED.
Same as CANCELLED
5.2.2  Idempotency Requirement
Shopify may deliver the same webhook multiple times. The handler must be idempotent. Before processing, check if the shopify_subscription_id and status combination has already been processed via the subscription_history table. If an identical event exists, return HTTP 200 without reprocessing.
5.2.3  Response Requirements
Webhook endpoint must return HTTP 200 within 5 seconds.
Processing that cannot complete in 5 seconds must be queued asynchronously.
Shopify will retry webhooks that do not receive HTTP 200 within 5 seconds.
Failed webhook processing must be logged with the full payload for manual recovery.
5.3  app/uninstalled — Handler Logic
Receive app/uninstalled webhook.
Mark store as inactive in stores table.
Check if active subscription exists.
If subscription is ACTIVE: note that Shopify cancels the subscription separately; await app_subscriptions/update CANCELLED webhook.
Do not delete any data. Set stores.is_active = false.
Log uninstall event to subscription_history.

SECTION 9  ·  Security Considerations
9. Security Considerations
9.1  Webhook Security
Every inbound Shopify webhook must be validated using HMAC-SHA256. The X-Shopify-Hmac-Sha256 header is compared against a server-side computed HMAC of the raw request body using the Shopify App Secret as the key.
Webhook endpoints must read the raw request body for HMAC validation before any JSON parsing. Parsing before HMAC validation creates a bypass risk.
Webhook validation failures must return HTTP 401 and be logged with IP, timestamp, and the first 32 bytes of the request for investigation.
Webhook endpoints must not be rate-limited by Shopify's standard rate limits but should implement their own DDoS protection at the infrastructure level via Vercel's edge network.
9.2  API Security
All billing API endpoints (except /api/billing/plans) require a valid Shopify session token validated server-side.
Session tokens must be validated on every request, not cached. Token expiry must be enforced.
Store isolation must be enforced at the query level. Queries must always filter by the authenticated store's ID. Cross-store data access must be impossible at the application layer.
The /api/billing/usage/ai-messages/increment endpoint must be called server-side only and must never be exposed to the client-side JavaScript bundle in a way that allows merchant manipulation of usage counts.
9.3  Data Security
StorePilot must never store, log, or transmit merchant payment card details, bank account information, or Shopify payment credentials.
The Shopify App Secret used for HMAC validation must be stored as an environment variable and never committed to source control or exposed in application logs.
All database connections must use TLS. Supabase connection strings must be stored as encrypted environment variables.
shopify_webhook_payload columns in subscription_history must be sanitised to exclude any personally identifiable information before storage if applicable.
9.4  Billing Integrity
StorePilot must never modify a subscription's billing amount or trial period outside of the Shopify API. Pricing is defined exclusively in the Shopify Partner Dashboard and passed through the API.
All subscription state changes must be sourced from Shopify webhooks or explicit Shopify API calls. StorePilot must never unilaterally change a subscription status to ACTIVE without Shopify confirmation.
The subscription_history table must be append-only. No application code may UPDATE or DELETE rows from this table. Database-level row security policies must enforce this.
9.5  Compliance
StorePilot must comply with Shopify's Partner Program billing requirements. Violation of Shopify billing terms may result in app removal from the Shopify App Store.
If StorePilot expands to non-Shopify billing in future, PCI-DSS compliance requirements must be assessed before implementation.
SECTION 10  ·  Future Billing Scalability
10. Future Billing Scalability
This section defines architectural considerations to ensure the V1 billing implementation can scale to accommodate future requirements without full rebuilds. These items are not V1 deliverables.
10.1  Usage-Based Billing Readiness
StorePilot's V1 billing is purely subscription-based. However, future monetisation models may include usage-based components — for example, charging per AI message beyond plan limits rather than hard-blocking. The billing_alerts and plan_entitlements tables should be designed to support overage configuration without schema changes. The ai_message usage tracking infrastructure built in V1 is the foundation for this.
10.2  Annual Billing Plans
Annual billing offers merchants a discount incentive and improves revenue predictability. Shopify's App Subscriptions API supports annual billing via interval: ANNUAL. V1 subscription creation should parameterise the billing interval to avoid hard-coding EVERY_30_DAYS. This allows annual plans to be introduced via configuration without code changes.
10.3  Multi-Platform Billing
The Founder Specification explicitly excludes WooCommerce, BigCommerce, and other platforms from V1. However, the billing abstraction layer should be designed to allow future payment providers (Stripe, Paddle) to be introduced without rewriting core subscription lifecycle logic. The subscriptions table should include a billing_provider column (defaulting to shopify) to support this future routing.
10.4  Per-Store Billing for Agency Tier
The Agency plan currently bundles 5 stores at a flat $199/month. Future iteration may introduce per-store pricing above the bundle threshold (e.g., $199 for 5 stores, $30/store beyond that). The plan_entitlements schema should accommodate an overage_price_per_unit field for this model.
10.5  Partner & Affiliate Billing
Shopify's Partner Revenue Share model may evolve. Future implementations may require tracking referral sources for revenue attribution. The subscriptions table should support a referral_code or partner_id foreign key for this purpose. No V1 implementation is required.
10.6  Promotional Pricing & Coupons
Shopify's App Subscription API supports custom trial periods and discount codes through partner agreements. The billing flow should accommodate a promo_code parameter in the subscribe endpoint that maps to extended trial periods or discounted first cycles when configured in the Partner Dashboard. V1 does not implement promotional pricing but must not architecturally prevent it.
10.7  Billing Dashboard for Agency Merchants
Agency plan merchants managing 5 stores will require a consolidated billing view showing per-store subscription status, data usage, and AI message consumption. The multi-store billing view should be designed as an extension of the existing entitlements endpoint structure rather than a separate billing system.
10.8  Automated Revenue Recovery
V1 relies on Shopify's default payment retry logic. Future iterations should implement proactive dunning management: automated email sequences to merchants with declining payments, in-app payment update prompts, and potentially a Shopify Partner Dashboard integration to monitor declined charges at scale.
